# Changelog

Version history and release notes for ModularIoT.

## Version Format

We follow semantic versioning: `MAJOR.MINOR.PATCH`

- **MAJOR** - Breaking changes
- **MINOR** - New features, backward compatible
- **PATCH** - Bug fixes, backward compatible

## Upcoming

Features and improvements in development:

- JavaScript, Python, and Java SDKs
- Advanced symptom patterns
- Enhanced notification channels
- Historical analytics dashboard

### `miot-calendar-client` v0.8.0 — Bloqueos de ventanas horarias (2026-05-02)

**`@microboxlabs/miot-calendar-client`** — [npm](https://www.npmjs.com/package/@microboxlabs/miot-calendar-client) | [GitHub](https://github.com/microboxlabs/modulariot/tree/trunk/packages/miot-calendar-client)

Agrega soporte tipado para distinguir ventanas horarias reservables de bloqueos no reservables.

**Actualizado — `client.calendars`:**
- `calendars.createTimeWindow()` y `calendars.updateTimeWindow()` aceptan `TimeWindowRequest.kind?: TimeWindowKind` — usa `WINDOW` para capacidad reservable o `BLOCK` para cierres
- `TimeWindowResponse.kind: TimeWindowKind` — las respuestas ahora identifican si cada ventana horaria es reservable o un bloqueo

**Nuevos tipos exportados:** `TimeWindowKind`

**`@microboxlabs/miot-calendar-client`** — [npm](https://www.npmjs.com/package/@microboxlabs/miot-calendar-client) | [GitHub](https://github.com/microboxlabs/modulariot/tree/trunk/packages/miot-calendar-client)

### `miot-calendar-client` v0.6.0 — Filtros de tareas por calendario (2026-05-02)

**`@microboxlabs/miot-calendar-client`** — [npm](https://www.npmjs.com/package/@microboxlabs/miot-calendar-client) | [GitHub](https://github.com/microboxlabs/modulariot/tree/trunk/packages/miot-calendar-client)

Agrega filtros de tareas a nivel de calendario y expone metadatos de provisionamiento de SlotManager para pantallas de configuración.

**Actualizado — `client.calendars`:**
- `CalendarRequest.filter?: CalendarFilter` — persiste filtros de tareas como `origin` y `destination` en un calendario
- `CalendarRequest.autoSlotManager?: boolean` — solicita provisionamiento de SlotManager por defecto al crear un calendario
- `CalendarResponse.filter?: CalendarFilter` — lee el filtro de tareas persistido desde las respuestas de calendario
- `CalendarResponse.hasSlotManager?: boolean` — consulta si un calendario tiene un SlotManager provisionado

**Nuevos tipos exportados:** `CalendarFilter`

### `miot-calendar-client` v0.3.0 — Gestores de Slots (2026-02-20)

**`@microboxlabs/miot-calendar-client`** — [npm](https://www.npmjs.com/package/@microboxlabs/miot-calendar-client) | [GitHub](https://github.com/microboxlabs/modulariot/tree/trunk/packages/miot-calendar-client)

Agrega un nuevo recurso de Gestores de Slots para automatizar la generación de slots. Cada gestor mantiene los slots generados un número configurable de días por adelantado y puede activarse manualmente o mediante un planificador externo.

**Nuevo — namespace `client.slotManagers`:**
- `slotManagers.list({ active? })` — listar todos los gestores de slots
- `slotManagers.create(body)` — crear un gestor para un calendario
- `slotManagers.get(id)` — obtener un gestor por ID
- `slotManagers.update(id, body)` — actualizar configuración (incluyendo reprocesamiento puntual)
- `slotManagers.deactivate(id)` — desactivar un gestor (eliminación lógica)
- `slotManagers.runAll()` — ejecutar todos los gestores activos sincrónicamente
- `slotManagers.run(id)` — ejecutar un gestor individual
- `slotManagers.listAllRuns({ limit? })` — listar ejecuciones recientes de todos los gestores
- `slotManagers.listRuns(id, { limit? })` — listar ejecuciones de un gestor específico

**Nuevos tipos exportados:** `SlotManagerRequest`, `SlotManagerResponse`, `SlotManagerRunResponse`

### `miot-calendar-client` v0.2.0 — Grupos de Calendarios (2026-02-19)

**`@microboxlabs/miot-calendar-client`** — [npm](https://www.npmjs.com/package/@microboxlabs/miot-calendar-client) | [GitHub](https://github.com/microboxlabs/modulariot/tree/trunk/packages/miot-calendar-client)

Agrega un nuevo recurso de Grupos de Calendarios que permite organizar calendarios en colecciones con nombre y filtrarlos o consultarlos por grupo.

**Nuevo — namespace `client.groups`:**
- `groups.list({ active? })` — listar todos los grupos
- `groups.get(id)` — obtener un grupo por ID
- `groups.create(body)` — crear un grupo
- `groups.update(id, body)` — actualizar un grupo
- `groups.deactivate(id)` — desactivar un grupo (eliminación lógica)

**Actualizado — `client.calendars`:**
- `calendars.list()` acepta el nuevo parámetro `groupCode` para filtrar calendarios por pertenencia a un grupo
- `CalendarRequest.groups?: string[]` — asigna grupos al crear o actualizar (`null` = sin cambios, `[]` = eliminar todos, `["code"]` = reemplazar todos)
- `CalendarResponse.groups?: CalendarGroupResponse[]` — los grupos a los que pertenece el calendario ahora se incluyen en la respuesta

**Nuevos tipos exportados:** `CalendarGroupRequest`, `CalendarGroupResponse`

### Arquitectura: Navegación Lateral Centralizada (2026-02-19)

**`apps/app` — cambio interno, sin impacto en la API**

La navegación de la barra lateral fue refactorizada para usar un único contexto compartido `SidebarNavigationProvider` por encima de los componentes de barra lateral de escritorio y móvil.

**Qué cambió para los operadores:**
- Los contadores de la barra lateral de escritorio y móvil ahora están garantizados para ser idénticos — anteriormente podían divergir porque cada barra lateral obtenía datos independientemente
- Los elementos de filtros de usuario bajo la sección de Tareas ya no se duplican después de navegar entre páginas

**Qué cambió para los desarrolladores:**
- Todos los datos de navegación (contadores, elementos de filtros de tareas dinámicos) se obtienen una vez por carga de página en lugar de dos veces
- La constante del módulo `pages.ts` ya no es mutada en tiempo de ejecución
- Las nuevas secciones con sub-elementos dinámicos pueden agregarse mediante `dynamicItemsSource` + un hook dedicado — ver [Extender la Navegación](/reference/frontend-architecture/extending-navigation)

Ver [ADR-001](/reference/architecture-decisions/adr-001-sidebar-navigation-context) para el registro completo de la decisión.
