# miot.metrics@1.0

La especificación formal del JSON Schema para payloads de métricas en ModularIoT. Esta es la referencia autoritativa para la validación del esquema.

## Resumen del Esquema

| Propiedad | Valor |
|-----------|-------|
| ID del Esquema | `https://modulariot.io/schemas/miot.metrics@1.0` |
| JSON Schema Draft | 2020-12 |
| Estado | Estable |
| Versión | 1.0 |

## JSON Schema Completo

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://modulariot.io/schemas/miot.metrics@1.0",
  "title": "Payload de Métricas MIOT",
  "description": "Estructura JSON canónica para payloads de métricas enviados a ModularIoT",
  "type": "object",
  "required": ["schema_version", "device_id", "timestamp", "metrics"],
  "properties": {
    "schema_version": {
      "const": "1.0",
      "description": "Identificador de versión del esquema"
    },
    "device_id": {
      "type": "string",
      "minLength": 1,
      "maxLength": 255,
      "description": "Identificador único del dispositivo"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "Timestamp ISO 8601 con zona horaria"
    },
    "metrics": {
      "type": "array",
      "minItems": 1,
      "items": { "$ref": "#/$defs/metric" },
      "description": "Array de mediciones métricas"
    },
    "asset_id": {
      "type": "string",
      "description": "Identificador de activo asociado (opcional)"
    },
    "sequence": {
      "type": "integer",
      "minimum": 0,
      "description": "Número de secuencia de mensaje para ordenamiento"
    },
    "batch_id": {
      "type": "string",
      "description": "Identificador de lote para agrupar payloads relacionados"
    }
  },
  "$defs": {
    "metric": {
      "type": "object",
      "required": ["key", "value"],
      "properties": {
        "key": {
          "type": "string",
          "pattern": "^[a-z][a-z0-9_]*(\\.[a-z][a-z0-9_]*)*$",
          "description": "Clave de métrica canónica (ej., engine.rpm)"
        },
        "value": {
          "oneOf": [
            { "type": "number" },
            { "type": "string" },
            { "type": "boolean" },
            { "type": "array" },
            { "type": "object" }
          ],
          "description": "Valor de la medición"
        },
        "timestamp": {
          "type": "string",
          "format": "date-time",
          "description": "Timestamp por métrica (anula timestamp raíz)"
        }
      }
    }
  }
}
```

## Referencia de Campos

### Objeto Raíz

| Campo | Tipo | Requerido | Descripción |
|-------|------|-----------|-------------|
| `schema_version` | string | Sí | Debe ser `"1.0"` |
| `device_id` | string | Sí | Identificador único del dispositivo (1–255 chars) |
| `timestamp` | string | Sí | Timestamp ISO 8601 con zona horaria |
| `metrics` | array | Sí | Array no vacío de objetos métrica |
| `asset_id` | string | No | Activo asociado (vehículo, equipo) |
| `sequence` | integer | No | Ordenamiento de mensajes (≥0) |
| `batch_id` | string | No | Agrupa payloads relacionados |

### Objeto Métrica

| Campo | Tipo | Requerido | Descripción |
|-------|------|-----------|-------------|
| `key` | string | Sí | Clave de métrica canónica |
| `value` | any | Sí | Valor de la medición |
| `timestamp` | string | No | Timestamp por métrica |

### Patrón de Claves

Las claves deben coincidir con el patrón regex:
```
^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*$
```

**Ejemplos válidos:**
```
engine.rpm
vehicle.speed
fuel.trim.short
engine.coolant_temp
x.vendor.custom_metric
```

**Ejemplos inválidos:**
```
Engine.RPM        ← mayúsculas no permitidas
engine-rpm        ← guiones no permitidos
123.metric        ← debe comenzar con letra
```

### Tipos de Valor

| Tipo | Tipo JSON | Ejemplo | Caso de Uso |
|------|-----------|---------|-------------|
| NUMBER | number | `2500`, `72.5` | La mayoría de las métricas |
| BOOLEAN | boolean | `true`, `false` | Estados (ralentí, freno, MIL) |
| STRING | string | `"1G1YY22G965109753"` | VIN, protocolo, identificadores |
| ARRAY | array | `["P0301", "P0420"]` | Códigos DTC |
| OBJECT | object | objeto | Freeze frames, datos complejos |

## Formato de Timestamp

Los timestamps deben ser ISO 8601 con zona horaria:

```
2024-01-15T10:30:00Z           ← UTC (preferido)
2024-01-15T10:30:00+00:00      ← UTC con offset explícito
2024-01-15T05:30:00-05:00      ← Local con offset
```

**Formatos inválidos:**
```
2024-01-15T10:30:00            ← Falta zona horaria
2024-01-15                     ← Falta hora
1705315800                     ← Timestamp Unix
```

## Ejemplos de Payloads

### Lectura Básica de OBD-II

```json
{
  "schema_version": "1.0",
  "device_id": "lector-obd-001",
  "timestamp": "2024-01-15T10:30:00Z",
  "metrics": [
    { "key": "engine.rpm", "value": 2500 },
    { "key": "vehicle.speed", "value": 72 },
    { "key": "engine.coolant_temp", "value": 85 },
    { "key": "fuel.level", "value": 65 }
  ]
}
```

### Con Asociación de Activo

```json
{
  "schema_version": "1.0",
  "device_id": "lector-obd-001",
  "asset_id": "vehiculo-flota-42",

  ]
}
```

## Validación

Los payloads se validan en este orden:

1. **Validación de esquema** — La estructura coincide con JSON Schema
2. **Validación de claves** — Las claves son canónicas o tienen prefijo `x.`
3. **Validación de tipos** — Los valores coinciden con tipos esperados
4. **Validación de límites** — Valores numéricos dentro de rango
5. **Validación de timestamp** — No muy antiguo ni en el futuro

Ver **Integración > Ingesta de Métricas > Reglas de Validación** para lógica completa de validación.
